# Deferred Overflow Review Analysis — 2026-07-08

## Analysis: Unresolved Context Overflow Fallback

### Overview

`packages/coding-agent` signals exhausted overflow recovery through `compaction_end` events carrying `unresolvedOverflow: true`. `packages/workflows` consumes that event in `StageSessionController`, converts it into a `WorkflowPromptModelFailure` with code `context_length_exceeded`, and relies on the existing model-fallback classifier to advance to the next candidate.

### Entry Points

- `packages/coding-agent/src/core/agent-session-auto-compaction.ts:43-72` — overflow auto-compaction detection and retry/exhaustion path.
- `packages/coding-agent/src/core/agent-session-auto-compaction.ts:185-258` — `_runAutoCompaction()` emits `compaction_end`.
- `packages/workflows/src/runs/foreground/stage-runner-controller.ts:329-333` — workflow session listener records unresolved overflow events.
- `packages/workflows/src/runs/foreground/stage-runner-controller.ts:353-393` — prompt/pause/resume loop consumes unresolved overflow.
- `packages/workflows/src/runs/foreground/stage-runner-unresolved-overflow.ts:3-15` — event-to-failure conversion.

### Core Implementation

#### 1. Signaling unresolved overflow

`_checkCompaction()` treats same-model context overflow as a compaction concern, not a normal retry (`agent-session-auto-compaction.ts:43-72`). On the second overflow recovery attempt for the same user message, it emits:

- `type: "compaction_end"`
- `reason: "overflow"`
- `willRetry: false`
- `unresolvedOverflow: true`
- terminal `errorMessage`

at `agent-session-auto-compaction.ts:50-60`.

`_runAutoCompaction()` also marks unresolved overflow when overflow compaction cannot produce a result or throws a non-abort failure. That flag is generated by `overflowUnresolved()` (`agent-session-auto-compaction.ts:181-183`) and emitted for missing model/result/failure cases at `agent-session-auto-compaction.ts:191-198`, `222-229`, and `242-253`.

#### 2. Consuming unresolved overflow in workflows

`StageSessionController.attachSession()` subscribes to all session events and updates `this.unresolvedContextOverflowMessage` whenever `unresolvedContextOverflowMessage(event)` returns a string (`stage-runner-controller.ts:329-333`).

`unresolvedContextOverflowMessage()` only matches `compaction_end` events with `unresolvedOverflow === true`; it uses `errorMessage` if present, otherwise a default message (`stage-runner-unresolved-overflow.ts:3-11`).

`throwUnresolvedContextOverflowIfPresent()` clears the stored message and throws `unresolvedContextOverflowFailure(message)` (`stage-runner-controller.ts:496-499`). That helper wraps the message in `WorkflowPromptModelFailure` with `{ code: "context_length_exceeded", stopReason: "error" }` (`stage-runner-unresolved-overflow.ts:13-15`).

The workflow fallback classifier treats `context_length_exceeded` as `request_incompatible` (`model-fallback-failures.ts:201-209`), and `request_incompatible` is fallbackable (`model-fallback-failures.ts:76-84`, `496-499`).

### Data Flow

1. Provider returns same-model overflow assistant message.
2. `_processAgentEvent()` runs `_checkCompaction()` after `agent_end` (`agent-session-events.ts:167-192`).
3. `_checkCompaction()` runs overflow compaction/retry (`agent-session-auto-compaction.ts:43-72`).
4. If recovery is exhausted, coding-agent emits `compaction_end` with `unresolvedOverflow: true` (`agent-session-auto-compaction.ts:50-60`).
5. Workflow listener stores the message (`stage-runner-controller.ts:329-333`).
6. `promptWithPauseResume()` calls `throwUnresolvedContextOverflowIfPresent()` after `activeSession.prompt()` resolves or throws (`stage-runner-controller.ts:370-381`).
7. The thrown `WorkflowPromptModelFailure` enters `handleCandidateFailure()` (`stage-runner-controller.ts:145-149`, `437-459`).
8. If retryable and not the last candidate, the current session is disposed and the loop advances (`stage-runner-controller.ts:448-458`).

### Reviewer Finding Assessment

#### P1: Late overflow bypass after deferred retry — valid

The deferred retry path can emit unresolved overflow after `StageSessionController.promptWithFallback()` has already treated the candidate as successful.

The key sequence is:

- Successful overflow compaction emits `compaction_end` without `unresolvedOverflow` and schedules continuation (`agent-session-auto-compaction.ts:237-238`).
- `_schedulePostAutoCompactionContinuationProbe()` uses `setTimeout(..., 100)` and later calls `_resumeAfterAutoCompaction()` when `willRetry` is true (`agent-session-auto-compaction.ts:140-159`).
- `_resumeAfterAutoCompaction()` fire-and-forgets `_runAgentContinue()` (`agent-session-auto-compaction.ts:166-174`).
- `AgentSession.prompt()` waits for `agent.prompt()`, `waitForRetry()`, and queued messages (`agent-session-prompt.ts:184-189`), but it does not wait for that later timer-driven continuation.
- `StageSessionController.promptWithPauseResume()` checks unresolved overflow immediately after `activeSession.prompt()` resolves (`stage-runner-controller.ts:370-371`) and then returns success (`stage-runner-controller.ts:379`).
- `promptWithFallback()` records the candidate as successful and returns (`stage-runner-controller.ts:143-144`).

If the timer-driven continuation later overflows and emits `unresolvedOverflow: true`, the workflow listener still records it (`stage-runner-controller.ts:329-333`), but the fallback loop for the original prompt has already exited. That is the bypass.

Concrete implementation guidance:

- Ensure overflow auto-retry completion is awaited by the same public `prompt()` call that initiated the overflow recovery, or expose a workflow-visible promise/state that `promptWithPauseResume()` can await before declaring the candidate successful.
- The critical invariant should be: for an overflow compaction with `willRetry: true`, `activeSession.prompt()` must not resolve as successful until the post-compaction continuation either succeeds cleanly or emits/throws the unresolved overflow terminal signal.
- If preserving the deferred listener-flush behavior is necessary, the continuation can still be deferred by a microtask/timer, but the session should retain an awaitable “post-compaction continuation” promise that `prompt()` or workflow `promptWithPauseResume()` observes before success.

#### P2: Pause before unresolved-overflow throw — valid

Controlled pause is designed to abort active work without finalizing the stage; the original prompt awaiter is held until resume (`stage-control-registry.ts:72-80`). The pause path records the stage paused and then calls `__requestPause()` (`executor-stage-control.ts:85-100`), which delegates to `StageSessionController.requestPause()` (`stage-runner-context.ts:207-208`). `requestPause()` creates `pauseRequest` and aborts the active session (`stage-runner-controller.ts:178-184`). Resume resolves that deferred pause request (`stage-runner-controller.ts:186-190`).

Inside `promptWithPauseResume()`, pause is honored before starting a prompt (`stage-runner-controller.ts:360-366`), after a successful prompt (`stage-runner-controller.ts:372-378`), and after a thrown prompt (`stage-runner-controller.ts:382-388`).

However, in the catch path, unresolved overflow is thrown before pause is inspected:

- `catch (err)` begins at `stage-runner-controller.ts:380`.
- `this.throwUnresolvedContextOverflowIfPresent()` runs first at `stage-runner-controller.ts:381`.
- only then does it read `pendingPauseAfterThrow` at `stage-runner-controller.ts:382`.

So if an abort-driven controlled pause and unresolved overflow are both present, fallback failure can win before the pause is held. That contradicts the controlled-pause contract.

Concrete implementation guidance:

- In the catch path, inspect and await `pauseRequest` before throwing unresolved overflow.
- After the pause resolves, consume unresolved overflow before continuing with any resume message, so the overflow signal is not lost by the next loop iteration resetting `this.unresolvedContextOverflowMessage` at `stage-runner-controller.ts:368`.
- Apply the same ordering principle to the success path if pause and unresolved overflow can coexist: pause should hold the awaiter first; unresolved overflow should be consumed after resume unless the resume exits without a message.

A safe ordering shape is:

1. If pause is pending, await resume.
2. If resume has no message, return paused/terminal scan state.
3. If unresolved overflow is present, throw it before sending another prompt.
4. Otherwise continue with the resume message.
5. If no pause is pending, throw unresolved overflow immediately.

### Key Patterns

- **Event bridge**: coding-agent emits `compaction_end`; workflow translates it into fallback failure (`agent-session-auto-compaction.ts:50-60`, `stage-runner-controller.ts:329-333`).
- **Stored terminal signal**: workflow stores unresolved overflow in `unresolvedContextOverflowMessage` until prompt control flow checks it (`stage-runner-controller.ts:23`, `496-499`).
- **Workflow-owned fallback**: fallback candidates are attempted in `promptWithFallback()` and failures are routed through `handleCandidateFailure()` (`stage-runner-controller.ts:107-150`, `437-459`).
- **Controlled pause/resume**: pause is represented by a deferred promise and session abort; resume resolves the deferred with an optional message (`stage-runner-controller.ts:178-190`).

### Configuration / Compatibility Notes

- No public API shape needs to change for the fallback signal: `AgentSessionEvent` already includes optional `unresolvedOverflow?: boolean` on `compaction_end` (`agent-session-types.ts:47-55`).
- `breaking_changes_allowed=false` points toward preserving the existing `prompt()` and event APIs while tightening internal await/ordering behavior.
- The fallback classifier already recognizes the generated `context_length_exceeded` code (`model-fallback-failures.ts:201-209`, `496-499`); the remaining work is delivery timing and pause ordering, not failure classification.
