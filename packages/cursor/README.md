# @bastani/cursor

First-party experimental Atomic provider for Cursor subscription models.

## Discovery, routing, and cache

Atomic uses Cursor's browser PKCE OAuth flow and private CLI-compatible HTTP/2 protocol. `POST /agent.v1.AgentService/GetUsableModels` is the sole authority for the authenticated account's runnable model existence, exact executable IDs, display data, and Max state. Atomic registers one model row for every returned GetUsable row in source order, preserving blank, whitespace-only, and duplicate IDs rather than filtering, deduplicating, or rewriting them. Raw textual provider/ID lookup selects the first exact occurrence without trimming, ambiguity rejection, suffix/context parsing, or rewriting. A later selected in-memory model object survives workflow validation and catalog refresh with that occurrence's current GetUsable Max/image routing; persisted provider/ID-only references intentionally restore the first occurrence. The selected ID is sent unchanged in both `ModelDetails.model_id` and `RequestedModel.model_id`.

`POST /aiserver.v1.AiService/AvailableModels` is a separate, best-effort metadata call for image capability only. Its decoder preserves explicit blank/whitespace identities, ordered duplicate variants, optional flags, and identityless parent rows without fabricating omitted strings. Atomic joins metadata to each GetUsable row through exact identity/variant evidence from the same authenticated account. Image input is enabled when one or more matching Available rows exist and every match explicitly reports image support; duplicate or distinct agreeing matches remain unambiguous, while any false or omitted match leaves the route text-only. AvailableModels cannot add, remove, reorder, or collapse executable rows, choose a route, supply Max state or request parameters, or block text usage. Model-family names are not used to infer image support.

Max comes exclusively from GetUsable and is encoded in both request model structures. `RequestedModel.parameters` is always empty. Atomic does not expand AvailableModels tuples, synthesize picker or backend routes, translate a reasoning selector into another route, or resolve legacy aliases, and it keeps no static list of historical Cursor IDs. Cursor-specific behavior requires the raw provider identity exactly lowercase `cursor`: workflow strings use provider-qualified `cursor/<raw-id>` bytes—including blank IDs and literal suffix/context-looking text—and saved/session/default/object or direct execution identities use provider `cursor`. Case or whitespace variants retain ordinary non-Cursor behavior and never activate Cursor discovery, errors, or authority; a bare (non-qualified) reference is likewise an ordinary non-Cursor value, and only an explicit lowercase `cursor/<id>` targets a Cursor route. An old or unavailable explicit Cursor ID in settings, a CLI command, a workflow, or a restored session fails clearly and must be reselected from the current authenticated catalog; Atomic does not substitute a nearest effort, static model, AvailableModels row, another Cursor model, or another provider.

Authenticated catalogs use schema-v3, account-scoped files named `~/.atomic/agent/cursor-model-catalog.json.account-<digest>` with a 30-minute TTL. They preserve the ordered GetUsable row sequence—including blank, whitespace-only, and duplicate IDs—plus optional same-account image flags. The digest is derived from the stable JWT subject claim, not from an OAuth token, and neither tokens nor account claims are persisted. Schema-v1, schema-v2, unscoped, and parameterized caches are ignored rather than migrated. A rotated token for the same account may reuse a fresh snapshot; another account cannot load or overwrite it. Per-account cache mutations are invocation-ordered: invalidation becomes visible to loads before its clear settles, later saves wait for earlier mutations, and a newer clear supersedes queued or active older saves without blocking another account. A later same-account save whose `fetchedAt` equals the stored record still wins because it is invoked last; only a strictly older timestamp is rejected. A fresh same-account v3 snapshot may keep its GetUsable-derived routes available during a temporary GetUsable failure, but stale snapshots, AvailableModels data, and static rows never become executable fallback.

Credential changes refresh immediately, while a current credential removal or lookup failure—during lifecycle discovery or execution authorization—revokes the active authority, unregisters Cursor rows, aborts prior leases/streams, and invalidates its scoped cache with a fixed sanitized login/reselection diagnostic. Stored and execution credential lookup invocations are identity-fenced before cache, catalog, authority, status, or discovery effects: once a newer lookup or disposal supersedes one that is still waiting, the older result is inert. Caller cancellation and provider disposal also detach pending first or second credential resolver waits; late settlement is observed but cannot start cache, discovery, registration, or transport effects, and an abort visible before Run invocation prevents transport from being called. Superseded or out-of-order requests cannot overwrite the latest account snapshot, and future-dated timestamps are not treated as fresh. Provider-bound execution uses a live revocable authority lease to recheck the supplied credential's opaque account scope, catalog generation, TTL, exact route, and selected current catalog occurrence synchronously at each irreversible transport or tool-result boundary. When the supplied scope is also the host credential manager's currently selected account, execution atomically removes the prior account's rows and joins that account's cache activation/GetUsable discovery; stale credentials cannot switch the provider back. Each lease schedules its own TTL expiry, which actively aborts silent active and paused streams, unregisters expired rows, clears only that scoped cache and freshness state, and requires rediscovery. Publishing or revoking a catalog, changing accounts, expiry, or disposing the provider invalidates the prior lease; inbound messages are rechecked after the read race before output, multi-result resumes recheck before every write, and aborted turns are removed. A removed, fabricated, expired, disposed, or non-current-account route is rejected before transport. A first-time `/login` succeeds only after authenticated discovery registers a usable catalog for that credential scope. Discovery participates in shutdown cancellation, its failures are redacted, `atomic --list-models` waits for a required refresh, and a successful live registration does not depend on best-effort cache persistence.

Low-level consumers constructing `CursorStreamAdapter` directly must provide `executionAuthorizer` or call `bindExecutionAuthority()` with their own current-catalog validator. Each authorization carries a live assertion and abort signal for one opaque published-catalog lease; the binding is one-shot and cannot be replaced. Custom code calling `CursorConversationStateStore` turn methods directly must propagate the authorized route and opaque turn handle through register, pause, resume, cancellation, and completion so stale cleanup cannot affect a replacement turn. Message iterators receive idempotent finalization and turns detach synchronously before potentially stalled transport cleanup. HTTP/2 Run streams release local codec and open-stream bookkeeping exactly once before awaiting remote close/cancel; adapter/session disposal starts cleanup, waits only for a bounded grace period, and detaches never-settling cleanup bookkeeping so a stalled transport cannot block provider shutdown. An unbound adapter rejects execution instead of trusting caller-supplied model compatibility metadata.

## Limitations

- Cursor is experimental. Its private API may change without notice, and use may conflict with Cursor terms or account policies.
- The rewrite intentionally breaks older experimental Cursor model IDs and cache formats. Open `/model` and reselect an exact route returned for the authenticated account.
- HTTP/2 requires bundled `@bastani/atomic-natives` for the current platform.
- The bundled credential path uses browser PKCE and Atomic's OAuth storage. Do not copy Cursor tokens into arguments, environment variables, logs, or proxy processes. Low-level adapters and injectable resolvers can supply credentials directly, so Atomic cannot prevent external code from placing a token elsewhere; enforced safeguards keep tokens/account claims out of the catalog cache, use private cache permissions, and sanitize known transport diagnostics.
- Only images on the current user turn and live mixed text/image MCP results are sent when exact same-account evidence advertises image input. Earlier-turn image blocks are omitted while their surrounding text/tool continuity remains. Image data must be non-empty canonical standard base64 after ASCII whitespace is stripped. The supplied MIME string is forwarded without allowlist validation; recognized image MIME types receive conventional filename extensions and other MIME values use `.img`.
- Reconstructing an image sent in an earlier turn and structured clipboard attachments remain follow-up [#1807](https://github.com/bastani-inc/atomic/issues/1807). Assistant-generated images are out of scope.
